# 聊天记录提纯示例库

## 示例 1：产品文案类

### 原始聊天记录

```
A: Hi Team, are you there? 对于新能源汽车落地页，老板说现在的版本太技术了，读起来像工程手册。
B: 对，用户不在乎千瓦时这些术语，他们想知道的是能不能带家人出去玩。
A: 没错！做成生活方式导向的，目标人群是家庭用户。提到后排宽敞和宠物友好材料。哦，控制在300字以内。分成"主标题"、"核心卖点"、"行动号召"三段。
B: 收到。像上个月我们喜欢的那套特斯拉 campaign 那样？
A: 对！那种感觉。快节奏、情感化、冒险感。
```

### 提纯后的 6-Scheme Prompt

```markdown
# 1. Role (角色与人设)
- 资深新能源汽车文案策划，专注家庭场景营销与生活方式品牌塑造。

# 2. Context (背景与意图)
- 品牌即将上线新款电动车，目标人群为现代家庭和宠物主人。现有营销物料过于技术化，无法触达消费者情感层，需转向生活方式优先的表达策略。

# 3. Task (核心指令与流水线)
- 撰写高吸引力的落地页文案，突出真实家庭场景（如公路旅行、露营）
- 聚焦用户利益点：车内空间宽敞、宠物安全的耐用内饰材料，而非原始技术参数

# 4. Output Format (输出格式)
分为三个明确段落：
- **[主标题]**：一句有力量的情感化标语
- **[核心卖点]**：3 个要点，聚焦家庭和宠物实用性
- **[行动号召]**：激发用户预约试驾的结尾句

# 5. Constraints (约束与红线)
- **长度**：全文不超过 300 字
- **语气**：冒险感、情感化、快节奏
- **禁用**：不得出现具体电池容量、电机参数等工程术语

# 6. Examples (Few-Shot 示例)
- **Input**: 家庭 SUV 生活方式文案
- **Output**:
  **[主标题]**
  打包行李。带上毛孩子。你的下一次冒险，安静启程。

  **[核心卖点]**
  - 全家都坐得下：为成长型家庭和周末出游设计的伸展级后排空间
  - 宠物友好的内饰：防刮划 vegan 真皮座椅，泥爪一擦即净
  - 露营就绪：车辆外放电接口直接为营地供电

  **[行动号召]**
  准备好重新定义家庭时光了吗？立即预约你的零排放试驾。
```

## 示例 2：需求文档类

### 原始聊天记录

```
张总: 那个积分系统，现在用户反馈说太复杂了
产品小李: 是的，规则太多，用户搞不懂怎么升级
张总: 简化！改成3个等级就够了。青铜、白银、黄金。
产品小李: 那升级条件呢？
张总: 消费满1000升白银，5000升黄金。青铜是注册就有。
产品小李: 积分有效期呢？现在是一年。
张总: 改成两年。另外，黄金会员要有点专属权益，比如免费配送、优先客服。
产品小李: 好的，我整理成需求文档。要包含什么？
张总: 等级规则、升级条件、权益对照表、迁移方案（老用户怎么映射到新等级）。给开发看的技术文档，别写营销话术。
```

### 提纯后的 6-Scheme Prompt

```markdown
# 1. Role (角色与人设)
- 资深 B2C 产品经理，精通会员积分体系设计与技术需求文档撰写。

# 2. Context (背景与意图)
- 现有积分系统规则过多导致用户认知成本高、升级路径不清晰。需将等级简化为 3 级（青铜/白银/黄金），并配套权益体系和老用户迁移方案，供开发团队直接落地。

# 3. Task (核心指令与流水线)
- 定义 3 个会员等级的升级条件与积分有效期规则
- 编制各等级权益对照表（含免费配送、优先客服等）
- 设计老用户到新等级的映射与迁移方案
- 输出面向开发团队的技术需求文档

# 4. Output Format (输出格式)
使用结构化技术文档格式：
- **1. 等级体系总览**（表格：等级名称 | 升级条件 | 积分有效期）
- **2. 权益对照表**（表格：权益项 | 青铜 | 白银 | 黄金 | 备注）
- **3. 老用户迁移方案**（编号列表：映射规则 + 数据处理步骤 + 灰度策略）
- **4. 技术实现要点**（编号列表：接口变更 | 数据迁移 | 灰度发布）

# 5. Constraints (约束与红线)
- **受众**：开发团队，纯技术语言，不含营销话术
- **等级**：仅 3 级（青铜/白银/黄金），不设更多层级
- **升级阈值**：消费满 1000 元升白银，5000 元升黄金，注册即青铜
- **积分有效期**：2 年
- **黄金权益**：必须包含免费配送和优先客服
- **格式**：表格 + 编号列表，不使用段落式描述

# 6. Examples (Few-Shot 示例)
- **Input**: 某电商平台 VIP 体系简化需求
- **Output**:
  **1. 等级体系总览**

  | 等级 | 升级条件 | 积分有效期 |
  |------|----------|-----------|
  | 青铜 | 注册即得 | 2 年 |
  | 白银 | 累计消费 ≥ 1000 元 | 2 年 |
  | 黄金 | 累计消费 ≥ 5000 元 | 2 年 |

  **2. 权益对照表**

  | 权益项 | 青铜 | 白银 | 黄金 | 备注 |
  |--------|------|------|------|------|
  | 免费配送 | ✗ | ✗ | ✓ | 黄金专属 |
  | 优先客服 | ✗ | ✓ | ✓ | 白银起享 |
  | 生日礼包 | ✗ | ✓ | ✓ | 含定制优惠券 |
  | 专属折扣 | ✗ | ✗ | ✓ | 每月 1 次 |
```

## 示例 3：会议纪要类

### 原始聊天记录

```
王姐: 今天产品评审会的纪要我发一下啊
- 首页改版：设计组这周五前出3版视觉方案，下周一内部评审
- 支付流程：技术说接口对接要2周，所以上线时间从6/15推迟到6/30
- 数据看板：李总要求加一个"用户留存"的周报模块，数据组负责
- APP启动速度：优化目标从2秒改到1.5秒，技术评估后下周五反馈可行性
刘哥: 补充一下，支付流程推迟的事要同步给运营，让他们调整推广节奏
王姐: 对，运营那边的push推送计划也要往后挪
```

### 提纯后的 6-Scheme Prompt

```markdown
# 1. Role (角色与人设)
- 资深项目管理助理，精通产品评审会议纪要的结构化整理与待办追踪。

# 2. Context (背景与意图)
- 产品评审会议讨论了 4 项议题（首页改版、支付流程、数据看板、APP 启动速度），产生了多项跨团队待办。需整理成结构化纪要并明确责任人与 deadline，同步给运营团队调整推广计划。

# 3. Task (核心指令与流水线)
- 按议题分组整理会议讨论要点
- 提取每项决议和待办，标注责任人与 deadline
- 标注需跨团队同步的事项（如支付推迟影响运营推广）
- 输出结构化会议纪要

# 4. Output Format (输出格式)
- **会议信息**：日期 | 参会人 | 主题
- **议题与决议**：按议题分节，每节含【讨论要点】【决议】【待办事项】
- **待办总览表**：表格（事项 | 责任人 | deadline | 状态）
- **跨团队同步项**：单独列表标注需同步的团队和内容

# 5. Constraints (约束与红线)
- **客观性**：仅记录讨论内容和决议，不加入个人推断
- **完整性**：4 项议题全部覆盖，不得遗漏
- **精准度**：deadline 用具体日期（如"6/30"），不用模糊表述（如"月底"）
- **待办格式**：每条待办必须含责任人和 deadline

# 6. Examples (Few-Shot 示例)
- **Input**: 产品评审会议 4 项议题的零散记录
- **Output**:
  **会议信息**
  - 日期：2026-06-29
  - 主题：产品评审会
  - 参会人：王姐、刘哥、李总、设计组、技术组、数据组

  **议题一：首页改版**
  - 决议：设计组输出 3 版视觉方案
  - 待办：设计组，本周五前完成 → 下周一内部评审

  **议题二：支付流程**
  - 决议：上线时间从 6/15 推迟至 6/30（技术接口对接需 2 周）
  - 待办：同步运营团队调整推广节奏和 push 推送计划

  **待办总览表**

  | 事项 | 责任人 | deadline | 状态 |
  |------|--------|----------|------|
  | 首页 3 版视觉方案 | 设计组 | 本周五 | 待启动 |
  | 视觉方案内部评审 | 相关人员 | 下周一 | 待启动 |
  | 支付接口对接 | 技术组 | 6/30 前 | 进行中 |
  | 支付推迟同步运营 | - | 尽快 | 待启动 |
  | 用户留存周报模块 | 数据组 | - | 待启动 |
  | APP 启动速度可行性评估 | 技术组 | 下周五 | 待启动 |
```
