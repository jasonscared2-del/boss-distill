# 6-Scheme Prompt 框架详解

## 概述

6-Scheme 是一种结构化 LLM Prompt 设计框架，将 prompt 拆解为六个正交支柱，确保生成质量的一致性和可复现性。

## 六柱详解

### 1. Role（角色与人设）

**目的**：为 LLM 设定高度具体的专家身份，激活相关领域知识。

**设计原则**：
- 使用"领域 + 级别 + 职能"三段式
- 避免泛化标签（"助手""专家""AI"）
- 优先使用行业认可的头衔

**优秀示例**：
- "资深新能源汽车文案策划，专注家庭场景营销"
- "敏捷产品经理，擅长 B2B SaaS 需求文档"
- "高级数据分析师，精通用户行为漏斗分析"

**反面示例**：
- "你是一个 AI 助手"
- "你是一个专家"
- "你是一个文案"

### 2. Context（背景与意图）

**目的**：提供业务上下文，让 LLM 理解"为什么做"而非仅"做什么"。

**必含要素**：
- 业务场景（产品上线、市场推广、内部汇报等）
- 市场痛点 / 用户痛点
- 目标受众画像
- 触发此任务的动因（老板要求、用户反馈、竞品压力等）

**设计原则**：
- 用 2-4 句话压缩背景，不写长篇大论
- 痛点要具体（"转化率低于行业均值 30%"）而非泛化（"效果不好"）

### 3. Task（核心指令与流水线）

**目的**：将任务拆解为可执行的 SOP。

**设计原则**：
- 按时间/逻辑顺序排列
- 每步以强动词开头：撰写、分析、生成、对比、筛选、优化
- 避免模糊指令："写好一点""注意质量"
- 量化目标："3 个方案""不超过 300 字""覆盖 5 个场景"

**优秀示例**：
```
1. 分析目标用户的核心痛点（3 个）
2. 针对每个痛点生成一个场景化文案
3. 对比三个方案的情感强度，推荐最优
4. 输出最终文案及推荐理由
```

### 4. Output Format（输出格式）

**目的**：确保输出结构可预测、可解析。

**常见格式**：
- **Markdown 表格**：对比类任务
- **JSON schema**：需要程序消费的场景
- **编号列表**：步骤、排名类
- **分段标签**：`[Hero Header]` `[Features]` `[CTA]`
- **混合格式**：表格 + 列表 + 文本块

**设计原则**：
- 明确每个段落的标题和内容类型
- 若有字数限制，标注在每个段落旁
- 示例：`[标题]（不超过 20 字）`

### 5. Constraints（约束与红线）

**目的**：设定硬边界，防止 LLM 越界。

**常见约束类型**：
| 类型 | 示例 |
|------|------|
| 长度限制 | "总输出不超过 300 字" |
| 语气要求 | "保持活泼有力，避免学术腔" |
| 禁用内容 | "不出现具体电池容量参数" |
| 强制包含 | "必须提及'家庭友好'关键词" |
| 数据范围 | "仅基于 2024 年后的数据" |
| 格式红线 | "不使用 emoji" |

**口语化翻译**：
| 用户原话 | 约束化表达 |
|----------|-----------|
| "别太枯燥" | 保持活泼语气，使用场景化描写 |
| "专业一点" | 使用行业术语，引用权威数据 |
| "短一点" | 总输出不超过 200 字 |
| "像 XX 那样" | 参考 [XX] 的风格特征 |

### 6. Examples（Few-Shot 示例）

**目的**：通过 Input/Output 对锚定 LLM 的输出预期。

**设计原则**：
- 至少提供 1 组 Input/Output 对
- Input 应贴近真实业务场景，不是占位符
- Output 应是理想输出，展示期望的质量、语气、格式
- 若对话中有参考案例，优先使用真实案例

**无参考案例时的生成策略**：
1. 从对话中提取业务领域
2. 构造该领域最常见的典型输入
3. 生成符合 1-5 柱约束的理想输出
4. 确保示例的质量足以锚定 LLM 的输出水平

## 框架使用检查清单

- [ ] Role 是否使用了精准人设（非泛化标签）
- [ ] Context 是否包含业务场景和痛点
- [ ] Task 是否拆解为强动词开头的步骤
- [ ] Output Format 是否指定了精确结构
- [ ] Constraints 是否覆盖了所有硬边界
- [ ] Examples 是否提供了可用的 Input/Output 对
- [ ] 最终 prompt 是否可直接输入 LLM 执行
